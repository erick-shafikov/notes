V8 выполняет js и WebAssembly. Основные стадии: парсинг и компиляция кода, исполнение, управление памятью

# стадии

задачи могут выполняться как в фоне так и в основном потоке

- Построение AST
- предварительный парсинг - Scanner (токенизация), Pre-parser (быстрый проход), Parser(Parser — построение полного AST) все стадии проходят в основном потоке
- Ignition — интерпретатор переводит в байт код, Ignition это register-based VM, задача запустить как можно раньше
- - байт код - промежуточный низкоуровневый код, который позволяет представить все операции в js для интерпретации на все виды машин. Раньше обходилось без него, сразу переводя в машинный код. Нужно что бы не писать каждое нововведение в языке для каждого отдельного процессора.
- Процесс AST интернализации - выделение в куче места для литералов (strings, numbers, object/array literals), создаются шаблоны (boilerplates)
- обработка явных подсказок Compile Hints, достигается с помощью специальных комментариев в коде (в зависимости от движка)
- Scanner and parser optimizations
- Финализация байт кода

# jit и его части

Компилятор движка v8 использует несколько уровней JIT компиляторов:

- Ignition - исполняет байткод
- Sparkplug - быстро компилирует байткод → машинный код почти без оптимизаций нужен для ускорения «холодного» кода, основная цель - кеш
- Maglev - меньше оптимизаций для warm code, некоторый байт код приводит к машинному исполнению
- TurboFan - когда функции и циклы вызываются много раз, агрессивные оптимизации, на выходе машинный код

# heap

Память JavaScript — показывает объём Heap по каждой вкладке

Данные и переменные используемые в js хранятся в куче heap, она делится на пространства:

- read-only-space - данные которые невозможно изменить типа false true, встроенные строки, internal objects, immutables
- new-space - v8 делит данные по поколения - старое и молодое. движок делает предположение, что те данные которые существую давно с меньшей долей вероятности будут удалены, new-space = young generation
- old-space - если данные молодого поколения переживают несколько сборок мусора, то отправляются сюда
- Code space - хранение исполняемого кода
- Large object space - данные превышающие лимит kMaxRegularHeapObjectSize
- New large object space - данные молодого поколения
- Old large object space
- Code large object space

Максимальный объем кучи варьируется по устройствам

# gc

Сборка мусора, опирается на механизмы:

- Minor Garbage Collector или Scavenger - реализует алгоритм Cheney, работает с полупространства From space и To space

Корни GC - глобальная область видимости JS или контекст отладчика. Проход переносит объекты из from space в to space, удаляются все кто остался в from space. Заполняется to space пока не переполнится

Основные моменты:

- Поколеническая сборка - объект сортируются по возрасту, новые как "nursery" и old generation
- Mark-and-sweep с уплотнением - переодически js останавливается что бы пометить достижимые объекты, есть stop-the-world фазы
- Инкрементальная сборка - то есть сборка происходит порционно
- Параллельная сборка - на многоядерных машинах
