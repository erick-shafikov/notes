# Получение данных страницы

Процесс браузера - это главный процесс, который отвечает за работу пользовательского интерфейса. HTML, CSS, JS грузит и парсит render process. Загрузка асинхронная и приоритетная, но браузер может распараллелить загрузку разметки с загрузкой js, css. HTML и CSS имеют высокий приоритет, скрипты средний - можно управлять async/defer. Изображения - низкий приоритет, можно менять с помощью link=preload. Так же используется preload scanner для параллельной обработки html, скриптов и css. Парсинг будет будет ждать загрузку скриптов (блокируют только sync-скрипты), так как в скриптах может находится запись в документ. Теги [async и defer](../html/tags/meta-tags.md#script) могут изменить это поведение.

# DOM

Токенизация и построение дерева, обработка блокирующих скриптов (без async и defer). Построение дерева может занимать весь поток main thread render процесса. Может помочь сканер пред-загрузки. async не будет блокировать выполнения построения dom, defer гарантирует порядок. CSS не блокирует парсинг HTML, но блокирует JS так как JS может использовать селекторы. Процесс построения DOM устойчив к ошибкам синтаксиса

# CSSOM

независимы друг от друга (DOM и CSSOM). CSSOM строится по мере парсинга. Специфичность применяется при вычислении стилей, а не при построении CSSOM. Повторное вычисление стилей (Recalculate Styles) - показатель общее время обработки стилей. Каждый браузер имеет свои предопределенные стили. Браузеры обычно ждут загрузки css перед выполнением js, HTML-парсинг продолжается. CSSOM строится параллельно DOM. CSS не блокирует HTML-парсинг, но блокирует выполнение JS и этап layout/paint.

# JS

js после загрузки компилируется, обрабатывается и исполняется. Преобразуется в AST, которое преобразуется в байт-код, который исполняется в основном потоке.

Браузер встречает script type=module src=main.js он рассматривает его как точку входа модуля. По мере импортов формируется граф зависимостей. Запуск каждого модулей осуществляется только после формирования графа зависимостей, который строится лениво. Поддерживается динамический импорт. При импорте из библиотек используется script type=importmap, это json-структура в которой указываются ссылки на библиотеки и алиасы.

# AOM

дерево доступности - пока не построится контент не готов для голосовых помощников. AOM строится на основе DOM и обновляется по мере его изменений. Используется скринридерами и assistive technologies.

# Многопоточность

Современные браузеры используют модель multi-process: browser, renderer, gpu, network и др. Процессы изолируются по origin (site isolation) и общаются через IPC.. Каждый iframe запускается в отдельном процессе. Процессы изолируются по site isolation. iframe может быть в том же процессе, если same-origin (Sandbox — per-process, Site isolation — per-site / origin). Каждый процесс запускает в отдельной песочнице для каждого сайта. Процессы общаются через IPC

Процессы браузера:

- Browser Process - центральный процесс, координирует остальные процессы, в ответственности - адресная строка, закладки, кнопка назад\впереде, сетевые запросы, доступ к файлам
- Renderer Process - процесс отрисовки, отвечает за то что внутри вкладки, задача превратить HTML, CSS и JS в веб страницу
- GPU Process - выполняет графические задачи, его используют все остальные для отрисовки на экране
- Network Process
- Utility Processes
- Plugin Process - отвечает ща плагины
- Extension Processes:

Каждый сайт занимает свой Renderer процесс, даже iframe-ы. Песочница обозначает что сайт запущен в изолированной среде от ОС

# Сравнения Chromium, Gecko и WebKit

CSS:

- Blink (Chromium): использует однопоточный движок стилей на C++ (основанный на WebKit)
- Gecko (Firefox): в рамках проекта Quantum (2017) Firefox интегрировал Stylo — новый CSS-движок на Rust с поддержкой многопоточности.
- WebKit (Safari): движок стилей WebKit, как и Blink, однопоточный (Blink отделился от WebKit в 2013 году, до этого архитектура была общей)

Графика:

- WebRender — рендере по умолчанию в Firefox
- Chromium использует гибридную модель CPU + GPU с активным GPU-композитингом.
- Safari (WebKit) формирует разметку через слои (называемые CALayer, поскольку на macOS и iOS используется система слоев Core Animation)

JS:

- Chromium, V8 - это Ignition, Sparkplug, TurboFan и Maglev
- Firefox, SpiderMonkey -интерпретатор, Baseline JIT, компилятор IonMonkey
- Safari, JavaScriptCore - LLInt, Baseline, DFG и FTL.

Процессы:

- Chromium - отдельный процесс для site-instance
- Firefox - число процессов настраиваемо
- Safari - site isolation, не строго 1 вкладка = 1 процесс
